Data Structures and Algorithms:

My understanding of algorithms was primarily done through the course CS-255 System Analysis and Design, it was where Big O notation, tree structures, and search algorithms was introduced. I only studied it on a surface level, and was only able to apply it to practical applications like simple database retrieval methods and my personal Travlr Getaway MEAN-stack project that I did in my full-stack course via MongoDB text indexes implementations to optimize queries from an O(n) to Log( n) complexity.

I‚Äôm aware that simply knowing binary search trees and hash tables exist is just a first step, what matters more is understanding why they‚Äôre useful and when and when not to use them, or where their strengths and weaknesses lie, outside of the curriculum I discovered bitmasking as a strong solution for managing multiple Boolean flags effectively.

bitmasking is a strategy to manage multiple true/false states in an optimized way. For example, in a video game setting with user permissions like (canEdit, canDelete, isAdmin), traditional implementations might store four separate Boolean fields for the permission table, but bitmasking can do all four stages with a single string: for example a file can be stored as 0b1010, which would loosely translate to canEdit=true, canDelete=false, canShare=true, isAdmin=false. Or in other words, checking a permission becomes a simple O(1) operation because it‚Äôs just scanning the location and binary option (0 or 1) or the string.

This technique would be particularly valuable in systems handling real-time state updates, such as multiplayer game servers tracking thousands of player states simultaneously, or role-based access control systems where permission checks occur on every API request. The memory efficiency and computational speed of bitwise operations make them ideal for high-throughput scenarios where traditional boolean fields or relational lookups would create bottlenecks. While I haven't implemented bitmasking in production code yet, understanding these advanced optimization patterns is just one useful tool in a sea full of other advance coding techniques, and there will always be room to learn more.

Security Mindset:

Security is supposed to be something that‚Äôs implemented into the system from the beginning, not bolted onto a project as an afterthought. It‚Äôs arguable whether or not I learned this a long time ago, but it‚Äôs a skill that many beginners have to consider when familiarizing themselves with code structures.

Implementing NoSQL injection protection in the project required understanding of how such attacks even work. The express-mongo-sanitize middleware prevents attacks by stripping MongoDB operators (like $ or &) from user inputs, but understanding why this matters requires grasping how those commands / operators / attacks even work. Similarly, protecting against regex injection attacks meant recognizing that even seemingly innocuous features like search boxes can introduce vulnerabilities if not properly secured.

The principle of defense-in-depth (implementing multiple security layers) rather than relying on a single safeguard is a reflection of common engineering practices. Sanitization middleware catches, injection attempts, schema validation ensures data integrity, and proper error handling prevents information leakage. No single measure if perfect, but common security practices build great muscle memory and general awareness of security measures and familiarity.
