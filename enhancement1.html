The Problem:

The original templates were designed for a static HTML with exactly three travel destinations. When the application was converted to use Handlebars templating and MongoDB for dynamic content and data addition / editing, a layout flaw in the websiteâ€™s UI appeared, adding more than three trips to the database (because 3 was the original default amount) caused a visual problem, Images began overlapping with the separator lines and the white space between them was inconsistent, and the appearance lost professionality.

The Solution:

Through debugging, file searching, and general understanding of the full-stack environment, I identified a few CSS issues that caused this problem.

  • Issue 1: Float Containment Failure: The images used float: right; to position themselves, but the parent <li> containers couldn't properly contain these floated children. This caused container height collapse and overlapping content as more items were added.

Solution: I added overflow: hidden; to #sites li, which created a block formatting context that forces the parent container to acknowledge and contain its floated children.

  • Issue 2: Inconsistent Padding Accumulation: The template applied 24px bottom padding to every <p> tag via #contents p. The original three static entries each had two paragraph tags, creating consistent spacing. However, new database entries with different description lengths accumulated different amounts of padding, resulting in uneven visual rhythm.

Solution: Removed the paragraph-level bottom padding and instead applied uniform 30px padding-bottom to the #sites li containers themselves, ensuring consistent spacing regardless of content length or number of paragraphs.

Another Problem:

I found another problem that started with a UI/UX problem. The Angular admin interface, started with a typo in trip-card.html where the Delete button used btnbtn-danger, which is a typo, it should be btn btn-danger, so it rendered as a bland looking black-and-white square. Angular Interface Before Enhancement:

Iâ€™m a huge fan of Material Design 3 principles, Iâ€™m no expert, but I like the idea of cleaning up an interface. My solution involved:

  • Button Styling: Replaced Bootstrap defaults with Material 3 guidance through the CSS. I added a higher radius (8px), subtle shadows, and letter-spacing for a polished look.
  • Color Hierarchy: Primary actions (Edit/Add Trip) use solid blue buttons, while destructive actions (Delete Trip) use outlined red buttons which mimic other professional designs from other websites.
  • Layout Consistency: I implemented flexbox across trip-listing.css and trip-card.css to ensure equal card heights and bottom-aligned buttons, eliminating awkward spacing caused by varying description lengths.

Angular Interface After Enhancement

The primary lesson was that designs optimized for static content often fail catastrophically when connected to dynamic databases. Scalable UI must be considered from the start or carefully retrofitted later. I learned how CSS cascades through component hierarchies in Angularâ€”making buttons align properly required understanding how flexbox properties flow through multiple DOM levels: Bootstrap columns, Angular component wrappers, card containers, and card bodies. Missing any link in this chain broke the layout.

I also gained practical experience with Material Design 3 principles, which provided a framework for intentional design decisions rather than arbitrary aesthetic choices. Understanding concepts like elevation, visual hierarchy, and spacing systems made the enhancement process more defensible and professional.

Challenges Faced

The primary technical challenge was overriding Bootstrap's opinionated defaults without affecting other parts of the application. Custom styles needed careful specificity management to take precedence appropriately.

The card alignment issue proved particularly frustratingâ€”conceptually simple (make cards the same height) but technically complex due to nested container behavior. Buttons floated wherever description text ended, requiring coordinated CSS changes in both trip-listing.css and trip-card.css.

There was also a learning curve in articulating why design choices matter beyond aesthetics. Inconsistent button placements create cognitive friction for users scanning multiple cards. Recognizing and addressing these UX concerns represents the difference between functional code and professional software.
